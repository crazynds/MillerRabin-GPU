# MillerRabinGPU — Project context for Claude

## What this project is

A CUDA-accelerated **Miller–Rabin primality test** for very large integers
(thousands of decimal digits). Numbers are represented as arrays of small
*limbs* (base `2^LIMB_BITS`, default 16 bits each) and all modular arithmetic
runs entirely on the GPU using NTT/FFT-based big-integer multiplication with
either Montgomery or Barrett modular reduction.

The intended use-case is hunting for primes of the form
`10^a - k·10^b - j·10^c - 1` (and related digit-reversed companions), but the
engine is fully general: any integer expressible as an arithmetic equation can
be tested.

## Key design decisions

- **Single binary target**: `bench_mr_gpu`. There are no tests binary, no MPI,
  no other executables — ignore the old prime_hunter references in git history.
- **Pluggable multiplication backend**: selected at compile time via `MUL_ALG`
  in `params.cmake`. Options: NTT merge (default), NTT 4-step, cuFFT, GPU-FFT,
  GPU-FFNT, schoolbook. All share the `Multiplier` type alias from
  `src/ops/mul/multiplier.cuh`.
- **Pluggable modular reduction**: `MOD_REDUCTION_ALG` — Montgomery (REDC) or
  Barrett. The reduction layer sits in `src/reductions/`.
- **Group-based short-circuit testing**: the input file may annotate lines with
  a group ID (`group_id: equation`). Equations in the same group are tested
  round-by-round; if one is composite the rest are skipped.
- **`params.cmake` is gitignored**: users copy `params.cmake.example`, which is
  tracked. `src/config.h` is also gitignored (generated by CMake from
  `src/config.h.in`).

## Input format (new, equation-based)

```
# comment
[group_id:] equation
```

- `equation` supports `+ - * / % ^` and parentheses. All integers are
  arbitrary-precision decimals. Example: `10^18001 - 25*10^1334 - 91*10^249 - 1`
- `group_id` is optional. Lines sharing a group_id are tested together:
  if equation 1 is composite, equation 2 is never tested.
- Lines without `:` are singleton groups.

The old format (five integers per line: `a b c k j`) is gone. `example.txt` has
been converted to the new format.

## Build

```sh
cp params.cmake.example params.cmake
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build -j
./build/bench_mr_gpu [--test] [--progress] [--report] [--config] example.txt
```

External dependencies fetched automatically by CMake:
- [Alisah-Ozcan/GPU-NTT](https://github.com/Alisah-Ozcan/GPU-NTT) — always
- [Alisah-Ozcan/GPU-FFT](https://github.com/Alisah-Ozcan/GPU-FFT) — only when
  `MUL_ALG` is `MUL_FFT_GPUFFT` or `MUL_FFNT_GPUFFT`

System requirements: CUDA ≥ 11, GMP (`libgmp-dev`), GCC ≤ 11 (for CUDA 11) or
`-allow-unsupported-compiler`.

## What to watch out for

- `n_limbs` must be the **same for all candidates in a GPU batch**. Groups with
  different equation sizes produce different `n_limbs` values; the driver
  sub-batches by `n_limbs` automatically.
- `LIMB_BITS` is a compile-time constant. Changing it requires a clean rebuild
  and affects every precision limit (see the table in `params.cmake.example`).
- The NTT merge backend is the default and most battle-tested. The FFT backends
  have double-precision rounding limits — the constructor checks them at runtime.
- `src/config.h` is generated; never edit it directly.
- `params.cmake` is gitignored; `params.cmake.example` is the tracked template.
